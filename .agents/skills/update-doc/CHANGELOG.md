# Changelog

## [1.9.0] - 2026-06-18

### Changed
- **TOC handling follows the new README-only policy.** The deterministic-fix rows for missing/plain/stale TOCs are removed — the README TOC is regenerated by the Step 7 generator (not a per-file LLM fix), and `docs/*.md` carry no TOC. Phase 3/4 no longer add or fix topic-doc TOCs; Phase 5 verifies only the README TOC via `--check`. `references/diagnostic-checklist.md` check 2 updated to README-only/generated. (Requires `nicolasdao/init-doc` ≥ 1.9.0.)

## [1.8.0] - 2026-06-15

### Changed
- **Phase 2 Step 4 maps the diff deterministically via `--affects`.** The diff→doc mapping now pipes `git diff --name-only` through the generator's new `--affects` mode (set membership computed in code — a deleted path still maps) instead of matching `source` globs against the diff by LLM judgment; the by-hand match remains the fallback when Python is unavailable. Phase 5 Step 7's orphaned-code check starts from the same call's `orphaned_paths` (then applies the existing prose-fallback and rename caveats). Requires `nicolasdao/init-doc` ≥ 1.7.0.

### Fixed
- **`skill.json` declares `systemDependencies`** (`git`, `python3`) — both are invoked (Step 3 diff + the generator in Steps 1a/4/7) but were previously undeclared.
- **`skill.json` now declares `authors` and `license` (`BSD-3-Clause`)** — aligning every constellation member on one license and author identity (the constellation previously mixed BSD-3-Clause, MIT, and none).
- **`allowed-tools` now includes `Agent`** — Phase 1 Step 1b spawns an Explore sub-agent for the judgment-only diagnostic, but `Agent` was undeclared, so the intended offload (keeping the corpus scan out of the master's context, per the scarce-context principle) could not run.

## [1.7.0] - 2026-06-14

### Added
- **Manifest regeneration at session end.** Phase 4 Step 7 now runs the manifest generator (`../init-doc/scripts/build-doc-manifest.py`) after all edits, rewriting `doc-manifest.json` and the README `<!-- BEGIN doc-index -->…<!-- END doc-index -->` managed block from frontmatter `description` in one deterministic pass — replacing the hand-maintained README doc-index verification. Phase 5 Step 8 adds a manifest-validity + `--check`-clean assertion.

### Changed
- **Phase 1 diagnostic split by producer.** The mechanical checks (line counts, TOC state, cross-link map, glob resolution, gotchas hub↔domain sync, README orphans) are now produced **deterministically by the manifest generator** (new Step 1a) and read straight from `doc-manifest.json`; the Phase 1 Explore sub-agent (Step 1b) is reduced to the **judgment** checks only — content staleness, numeric-claim verification, stale file/feature references (checklist check 5). This removes a full sub-agent read pass for findings a script computes exactly, and keeps the master's context lean. Falls back to the legacy full-checklist sub-agent run when Python is unavailable.
- **`references/diagnostic-checklist.md`** now marks each check as 🛠 script-produced (checks 1, 2, 3, 4, 6) or 🧠 sub-agent/judgment (check 5), with a table mapping each mechanical check to the manifest field that supplies it.
- Phase 2 Step 4 and Phase 5 Step 8 coverage checks now read the SOURCE MAP / `source_unresolved` / `dangling` from the manifest rather than a sub-agent-assembled diagnostic report.

## [1.6.0] - 2026-06-14

### Added
- **Deterministic diff→doc mapping via `source` frontmatter.** Phase 2 Step 4 now maps changed files to the docs they affect by **set membership** against each doc's `source` globs (from the new diagnostic SOURCE MAP) instead of the prior "could this doc be affected?" LLM judgment. A prose-judgment fallback is retained for docs that declare no `source`, so partially-migrated and legacy corpora keep working. (Schema: `nicolasdao/init-doc` `references/standards.md § Frontmatter`.)
- **Diagnostic check 6 — Frontmatter and Source Mapping** in `references/diagnostic-checklist.md`: the Phase 1 sub-agent now extracts each doc's `source` globs and whether they resolve, returning a SOURCE MAP in the report so the master agent never re-reads the corpus.
- **Coverage check** (Phase 5): live-glob repair, corpus-wide **dangling-doc** detection (a doc whose `source` no longer resolves), and **orphaned-code** detection (changed paths no doc covers) — surfaced as recommendations, never auto-applied. Guarded against partial-migration and rename add-side false positives.
- **Source-aware writes:** new gotchas domain files are created with `source` frontmatter, and `source` globs are kept current when a change renames, moves, or deletes documented code.

### Changed
- Standards enumeration in SKILL.md now includes `frontmatter`.

## [1.5.1] - 2026-05-25

### Changed
- skill.json description trimmed to fit the 200-char soft cap (implementation-detail clauses removed). Namespace consistency: "Project Memory" → "ProjectMemory".

## [1.5.0] - 2026-05-19

### Changed
- SKILL.md frontmatter description refactored to the five-slot grammar with `ProjectMemory —` Domain anchor, explicit `Use when` triggers, and `Not for` negative redirecting to sibling skills (`init-doc`, `refactor-doc`). Length reduced from 558 chars (well over the 250-char soft cap) to ~245 chars. Auto-invocation routing signal significantly strengthened.
- skill.json description aligned with the Project Memory constellation positioning.
- Monolithic gotchas refactor suggestion now points to the new `/refactor-doc` skill (Phase 3 gotchas routing rules and `references/diagnostic-checklist.md`). Previously pointed to `/init-doc refactor`, which has been removed in init-doc 1.0.0.

### Positioning
- This skill is now formally positioned as the **maintain** satellite of the forthcoming `nicolasdao/project-memory` constellation.

## [1.4.0] - 2026-05-19

### Added
- Dependency on `nicolasdao/init-doc` — formalizes the shared standards reference relationship between the two skills.
- Standards section pointing to the canonical `../init-doc/references/standards.md` (single source of truth for size guardrails, TOC, cross-linking, gotchas structure, writing standards, protected files, project root determination).
- Precedence exception clarification: user notes never override the Protected Files rule. Closes a previously ambiguous boundary between "user notes take precedence" and "protected paths are inviolable."

### Changed
- Step 0 (Project Root) now references the shared procedure in `standards.md` instead of inlining the rule, ensuring consistent project-root resolution across the whole skill family.
- The "Documentation Standards Reference" table and "Protected Files (NEVER modify)" section have been removed in favor of a single Standards Reference pointer to the canonical file. The rules are unchanged; only their location has consolidated.

## [1.3.0] - 2026-04-08

### Added
- Gotchas hub+domain architecture support: detects `docs/gotchas/` directory for hub+domain format vs monolithic format
- Gotchas routing rules in Phase 3: routes new gotchas to the correct `docs/gotchas/<domain>.md` file, creates new domain files when needed
- Deterministic hub sync: after any domain file change, regenerates `docs/gotchas.md` hub from domain file headings (full regeneration, not partial patch)
- Monolithic fallback: if no `docs/gotchas/` directory exists, adds gotchas to `docs/gotchas.md` directly and suggests `/init-doc refactor` if over 750 lines
- Diagnostic checklist check #4 expanded to structured 4-step gotchas audit (format detection, hub check, integrity check, size warning)
- Gotchas hub size guardrail (50 lines) in diagnostic line count check and validation
- Phase 5 gotchas integrity validation: hub exists, hub in sync with domain files, no oversized domain files

### Changed
- Sub-agent scan glob now includes `docs/gotchas/*.md` alongside `docs/*.md`
- Deterministic fixes table expanded from 1 gotchas row to 3 (hub missing, directory missing, hub out of sync)
- Diagnostic aggregate report GOTCHAS STATUS expanded to multi-field format (format, hub, domain dir, sync issues, etc.)

## [1.2.0] - 2026-04-02

### Added
- 5-phase workflow (Diagnostic, Change Analysis, Plan, Execute, Validate) replacing flat 6-section structure for more robust orchestration
- Sub-agent diagnostic phase — spawns an Explore agent to audit all doc files against init-doc standards without consuming master context window
- Deterministic compliance layer — TOC, cross-links, size guardrails, and gotchas.md existence are fixed unconditionally, separate from LLM judgment calls
- Size guardrails aligned with init-doc (README.md at 750 lines, docs/ files at 750 lines) with split-before-writing rule
- TOC enforcement — adds linked markdown TOC (with anchor links) if missing, converts plain-text TOC to linked format, regenerates stale TOCs
- docs/gotchas.md awareness — diagnostic checks for existence, creates if missing, populates with non-obvious behavior discovered during updates
- Documentation Standards Reference table — codifies writing quality standards (audience, examples, terminology, single-concern) aligned with init-doc
- Diagnostic checklist reference file (references/diagnostic-checklist.md) for structured sub-agent audits

### Changed
- File creation threshold now uses size guardrails instead of arbitrary 50-line heuristic — files over 750 lines must split, files approaching the limit split before writing
- Cross-link checking expanded from new-files-only to all documentation files via diagnostic audit
- Structural splits now happen before content writes to prevent post-hoc reorganization

## [1.1.0] - 2026-04-02
