# Changelog

## [0.8.0] - 2026-06-18

### Changed
- **Follows the new README-only TOC policy.** "Fix TOCs" becomes "Strip legacy TOCs" — refactor-doc removes any in-file `## Table of Contents` from `docs/*.md` and gotchas domain files (a TOC is generated scaffolding, not content, so removal is not a content change); the README TOC is regenerated by the generator. New domain files and split products are written without a TOC. The `TOCs drift` trigger is dropped from the description and the "DO use" list — there are no stale TOCs left to fix. (Aligned with `nicolasdao/init-doc` ≥ 1.9.0.)

## [0.7.0] - 2026-06-15

Clears the mega-skill soft-cap warning — refactor-doc owns a single verb cluster (restructure), so compression is the correct remedy, not Constellation decomposition.

### Fixed

## [0.6.0] - 2026-06-14

### Added
- **Re-architect (deep) mode — holistic, source-aware graph re-derivation.** refactor-doc now has two modes. The default **structural** mode is unchanged (source-blind, redistributes existing content into a better arrangement of the *existing* domains). The new **re-architect** mode handles the long-lived-project problem: when the documentation's domain decomposition has gone *archaic* against the code (a monorepo grown over months/years whose week-2 boundaries no longer match the source), it reads the **source structure** (not prose) to re-derive a better domain set, then migrates the existing content **verbatim** into the re-shaped graph. Wired through all phases: Phase 1 builds a code-vs-docs **divergence map** from the source structure; Phase 2 proposes the re-derived decomposition + the full old→new content map + a list of undocumented-subsystem **gaps**, and gets explicit user approval for the mode; Phase 3 migrates content verbatim, may re-derive `source` globs to the new boundaries, and writes **gap markers** (never authored prose) for undocumented subsystems; Phase 5 review proves no content was lost.
- **Undocumented-subsystem gap flagging.** Where re-architecture exposes source with no existing documentation, refactor-doc flags it as a gap for `init-doc`/`update-doc` and writes at most an explicit placeholder — it never authors content from code. The gap list is surfaced in the summary.
- **Explicit mode invocation.** `$ARGUMENTS` may begin with a mode word — `/refactor-doc re-architect [notes]` or `/refactor-doc structural [notes]` (synonyms: `deep` / `rearchitect`, `shallow`) — which selects the mode and is consumed; the rest is notes. A bare `/refactor-doc` keeps the default (structural, auto-escalate-and-confirm). An explicit choice is honored but judgment still surfaces a mismatch (asked-to-re-architect-but-structure-is-sound → don't manufacture churn; asked-for-structural-but-graph-is-archaic → do it but flag that a re-architecture is warranted). `argument-hint` updated to `[structural|re-architect] [notes]`. Pairs with the new `project-memory` Inspect drift detector, which can hand you a direct `/refactor-doc re-architect` recommendation.

### Changed
- **The protected invariant is now stated as *content-preservation*, not *source-blindness*.** refactor-doc's real promise — "it never authors or rewrites prose from code; it only changes where your words live" — holds in every mode; "source-blind" is just the structural mode's default. Source is read in exactly two scoped, structure-only cases (frontmatter-glob backfill, and re-architect's domain analysis), never to write or fact-check content. Identity, "When to Use," the review-criterion note, the Frontmatter Backfill framing, and the Constraints are all rewritten around this. The Phase 5 verbatim set-equality criterion is unchanged and still applies in re-architect mode (gap markers + re-derived scaffolding are explicitly allowed, not fabrication).
- skill.json description rewritten to reflect re-architecture and brought under the ~200-char soft cap (the prior "Source-blind except…" wording was now inaccurate).

## [0.5.0] - 2026-06-14

### Added
- **Manifest regeneration after restructuring.** Phase 3 gains step 9: after all splitting/redistribution is written (still inside the backup/review flow), run the manifest generator (`../init-doc/scripts/build-doc-manifest.py`) to rewrite `doc-manifest.json` and the README `<!-- BEGIN doc-index -->…<!-- END doc-index -->` managed block from frontmatter — so new split products and the hub+domain structure are indexed and linked without hand-editing. Phase 4 gains step 7: manifest validity + `--check`-clean assertion.
- **Group-boundary guard (monorepos).** Phase 3 step 5 forbids relocating content or repartitioning `source` globs across a documentation **group** boundary during a split (a split product stays in its parent's group); if the only sensible split would straddle two sub-projects, refactor-doc stops and surfaces it. Phase 4 step 7 confirms every split product shares its parent's `group` in the manifest. (See `../init-doc/references/standards.md § Documentation Manifest`.)
- **Legacy-corpus robustness.** Phase 1 inventory now records whether `doc-manifest.json` (and the README managed block) exists. Phase 2's decision gate treats a **missing manifest or docs lacking frontmatter as a modernization gap that blocks a clean exit** — a structurally-sound but unmodernized legacy corpus gets a *minimal modernization pass* (backfill frontmatter + generate the manifest, skipping unneeded structural steps) instead of "nothing to do." Phase 3 step 8 **guarantees a `## Documentation` section exists** (migrating a legacy README's differently-named doc list into one) so the generator has somewhere to place the managed block — without it the generator writes nothing, not even the manifest; step 9 adds the recovery instruction for that stop and notes the manifest is created from scratch on first run.

### Changed
- Phase 3 step 8 no longer hand-adds README link entries for new docs/ files — that list is the generated managed block (step 9); step 8 now updates the section's human narrative **and ensures a `## Documentation` section exists** to host the block.

## [0.4.0] - 2026-06-14

### Added
- **Frontmatter backfill for legacy docs — the one bounded exception to source-blindness.** The trigger is a doc with **no frontmatter block at all** (a legacy doc predating the feature) — *not* a doc that merely lacks `source`. A present block, even one with `description`/`tags` and no `source`, means the `source` choice was already made deliberately (e.g. `docs/mission.md`, which documents intent and carries no `source` by design); such docs are left untouched. For a true no-frontmatter doc, refactor-doc creates the block: `description`/`tags` from the doc's content (fully source-blind), and `source` by reading the codebase via the **same methodology `init-doc` uses** — so backfilled globs are first-class, not second-rate guesses (and source-exempt or abstract docs correctly receive no `source`). Wired through all phases: Phase 1 inventories frontmatter presence and flags no-frontmatter docs; Phase 2 surfaces the proposed globs in the user-approved plan; Phase 3 step 6 performs the backfill; Phase 4 step 6 verifies every backfilled glob resolves and is not a catch-all.
- **Shared `source`-derivation procedure** extracted to `nicolasdao/init-doc` `references/source-mapping.md`, referenced by both init-doc (normal operation) and refactor-doc (the backfill exception only). Keeps the two skills deriving globs identically.
- **New `## Frontmatter Backfill` section** documenting the trigger, what is backfilled and how, the hard scope guard, and the abstract-doc carve-out.

### Changed
- The absolute "NEVER read source code" rule is now scoped: source-blind by default, with the single bounded exception of deriving missing `source` globs. **Hard scope guard:** even under the exception, source is read ONLY to map globs — never to rewrite, verify, or fact-check doc prose; the "preserve content verbatim" invariant remains absolute.
- Phase 5 review note clarifies that backfilled frontmatter is new metadata (exempt from the verbatim-set-equality "no fabrication" check), while its globs are still validated to resolve.
- Constraints add: never read source to alter content under the exception; never invent a glob to fill the field (leave absent for abstract docs); every backfilled glob must resolve.

## [0.3.0] - 2026-06-14

### Added
- **Backup-and-review safety protocol.** refactor-doc no longer mutates the corpus in a single irreversible big-bang. A new shared reference, `nicolasdao/init-doc` `references/doc-review.md`, defines a three-part protocol now wired into Phases 2–5: (1) a **decision gate** where a clean exit ("docs already in good shape, no change") is a valid result; (2) a **backup** of `README.md` + `docs/` to `.project-memory-backup/` at the project root, taken in Phase 3 *before* any write; (3) a **scrutinize-derived review loop** (new Phase 5) that proves the new tree against the backup.
- **Phase 5: Review & Cleanup.** A read-only cold-reader sub-agent (given the backup, the new tree, and the criterion but *not* the skill's reasoning) plus the author's own simple+harvest passes find content loss, fabrication, and structural breakage; findings are reconciled (disagreements weighted heaviest), **proven against the backup via grep before any fix** (unprovable → dropped), fixed (proven auto-applied, risky surfaced), and confirmed by re-running the Phase 4 structural battery. The loop repeats until lenses are exhausted (hard cap 3 passes), not until "certainty." On clean convergence the backup is deleted; otherwise it is retained with the unresolved findings surfaced. This makes the long-standing "no content lost" guarantee **provable** instead of aspirational — previously the original was destroyed before any check could run against it.
- **Review criterion: verbatim set-equality.** Because refactor-doc only redistributes content, the review requires backup ⊆ new (no loss) and new ⊆ backup (no fabrication), modulo generated scaffolding (hub text, regenerated TOCs).
- `Agent` and `AskUserQuestion` added to `allowed-tools` (cold-reader sub-agent; surfacing risky fixes).

### Changed
- Phase 2 (Plan) now opens with the decision gate — a clean "no restructure needed" exit is explicitly valid.
- Phase 3 (Execute) gains step 0: take the backup before creating, moving, or rewriting any file.
- Backup directory `.project-memory-backup/` registered as a transient excluded path in `references/standards.md § Protected Files`.

## [0.2.0] - 2026-06-14

### Added
- **`source` frontmatter preservation across splits.** When Phase 3 splits an oversized `docs/*.md`, every split product now carries frontmatter: the original `source` globs are repartitioned across the products by best-effort match to the content that moved, and kept on **all** products when a glob's home is unclear (over-coverage is safe; dropping a glob silently breaks `update-doc`'s diff→doc mapping). Because refactor-doc does not read source code, it only redistributes existing globs — it never invents new ones. (Schema: `nicolasdao/init-doc` `references/standards.md § Frontmatter`.)
- **Plan-gate disclosure** (Phase 2): the proposed `source` repartition for any split file is now listed in the plan the user approves, so the doc→code mapping redistribution is never done silently in execute.
- **Validation** (Phase 4): a new check confirms the union of a split's products covers every `source` glob the original declared — no glob may be dropped.

### Changed
- Standards enumeration in SKILL.md now includes `frontmatter`.
