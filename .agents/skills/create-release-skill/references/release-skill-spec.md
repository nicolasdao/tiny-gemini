# Release Skill Specification

Every release skill generated by create-release-skill MUST implement these behaviors. This is the contract that all generated skills share regardless of project type, language, or deployment strategy.

### Portability Rule

All paths in a generated release skill MUST be relative — relative to the project root, the git root, or using `${CLAUDE_SKILL_DIR}` for files bundled with the skill. Never embed absolute paths (e.g., `/Users/someone/projects/...`). Generated skills must work on any machine that clones the repository.

---

## 1. Arguments

Every generated release skill uses:

```yaml
arguments: [action, note]
argument-hint: "[patch|minor|major|draft|auto] [\"description\"]"
```

### $action (optional, position 0)

| Value | Behavior |
|---|---|
| `patch` | Explicit patch bump, full release |
| `minor` | Explicit minor bump, full release |
| `major` | Explicit major bump, full release |
| `draft` | Mode C — write unreleased notes only, no version bump or tag |
| `auto` | AI determines bump (use when $note is also provided) |
| omitted | Same as `auto` |

### $note (optional, position 1)

Quoted description to include in the release. When provided:
- MUST be reflected in changelog entries
- MUST be factored into the bump decision (e.g., if the note describes a breaking change, bump major)
- Is additive to git analysis, not a replacement

---

## 2. Release Modes

### Mode A — Hot Release

The current session contains substantial work on the target project.

**Detection**: The AI introspects on its own conversation. If it has participated in meaningful work on the target project during this session (file edits, bug discussions, feature implementation, code reviews), this is Mode A.

**Behavior**: Uses BOTH session context AND git analysis. Changelog quality is highest because the AI knows intent, trade-offs, and reasons behind changes — not just what the diff shows.

### Mode B — Cold Release

New session with no prior context for the target project.

**Detection**: The session is thin — just started, or work was on a different part of the codebase.

**Behavior**:
1. Read current version from the version file
2. Read CHANGELOG.md for the last release entry
3. Find the last release tag matching the project's tag format
4. Run `git log <last-tag>..HEAD -- <project-path>` to find changes
5. If no tags exist, use all commits touching the project path
6. For unclear commits, examine actual diffs (`git diff <last-tag>..HEAD -- <project-path>`)
7. Check for existing `[Unreleased]` notes in CHANGELOG.md (from a previous Mode C draft)
8. Before classifying, pause and ask the user: "I don't have session context for this project. Here's what I found from git. Is there anything the commits don't capture — intent, trade-offs, or context I should know?"

### Mode C — Draft (Context Snapshot)

User passed `draft` as $action.

**Behavior**:
1. Analyze changes using the same process as Mode A or B (depending on session richness)
2. Write or update the `## [Unreleased]` section in CHANGELOG.md with classified change notes
3. Do NOT bump version, create tags, or modify the version file
4. Stage CHANGELOG.md
5. Commit with message: `docs(<project-name>): update unreleased changelog notes`
6. Show the user what was written

**Pre-flight exception**: Mode C relaxes the clean-directory check. Uncommitted changes are expected because the user is mid-work. Only verify that CHANGELOG.md itself is not in a conflicting state (e.g., unstaged changes to CHANGELOG.md that would be overwritten).

**When a full release later finds [Unreleased] notes**: Use them as a starting point but cross-reference against git to catch anything added after the draft. The draft is a head start, not a contract — new commits since the draft must also be captured.

---

## 3. Pre-flight Check

**This is a hard gate for Mode A and Mode B. The skill MUST NOT offer a "proceed anyway" option.**

**Mode C exception:** Section 2 (Mode C — Draft) relaxes the clean-directory check because the user is mid-work. See Mode C's "Pre-flight exception" note below.

Release skills only commit release-specific files (version file + CHANGELOG.md). They do NOT commit feature/fix code. If there are uncommitted changes in the project directory, the release produces a tag whose commit does not contain the changes it ships — a silent, high-blast-radius bug.

### Check

```bash
git status --porcelain -- <project-path>/
```

### If non-empty, abort immediately

```
Cannot release — uncommitted changes in <project-path>/:

<git status output, verbatim>

The release only commits version file + CHANGELOG.md.
It does NOT commit your feature/fix code, so running it now
would produce a tag that doesn't contain the changes it ships.

Commit your changes first, then re-run the release skill.
```

### Exception

Mode C (draft) skips the full clean-directory check. See Mode C section above.

## 4. Change Analysis

### Source of Truth Priority

1. **Session context** (Mode A only) — highest quality, understands intent
2. **$note argument** — explicit user-provided description, always respected
3. **Existing [Unreleased] notes** in CHANGELOG.md — from a previous Mode C draft
4. **Git log** — commits since last release tag, filtered to project scope
5. **Git diff** — when commit messages are unclear, examine actual file changes

### Finding the Last Release

1. Read the version file for the current version
2. Look for git tags matching the project's tag format (e.g., `v1.2.0` or `frontend-v1.2.0`)
3. If both exist and disagree, trust the version file — tags may be missing or out of sync
4. If no tags exist at all, diff against the initial commit or use all commits touching the project path

### Filtering

- Only consider commits touching files within the project's directory scope
- Ignore merge commits unless they contain meaningful context in the message
- Squash related commits into single logical changes (one bullet per logical change, not per commit)

### Nothing Meaningful Changed

If analysis determines nothing meaningful changed since the last release, inform the user:

```
There's nothing worth releasing. Changes since the last release are limited to:
<brief description — e.g., "CI config tweaks and internal refactors">

Would you like to proceed with a patch anyway, or skip the release?
```

Use AskUserQuestion with options to proceed or abort.

## 5. Change Classification

Use [Keep a Changelog](https://keepachangelog.com) categories:

| Category | Description | Default Bump Signal |
|---|---|---|
| **Added** | New features, endpoints, pages, capabilities | minor |
| **Changed** | Changes to existing behavior, UI, or API contracts | minor or patch |
| **Deprecated** | Features marked for future removal | minor |
| **Removed** | Features, endpoints, pages, or capabilities removed | major |
| **Fixed** | Bug fixes | patch |
| **Security** | Vulnerability fixes | patch or minor |

---

## 6. Bump Rules

| Condition | Bump |
|---|---|
| Breaking change (removed features, changed API contracts, incompatible changes) | **major** |
| New feature, capability, endpoint, page, or integration | **minor** |
| Bug fixes, performance improvements, internal refactors, dependency updates, docs | **patch** |
| Nothing meaningful changed | **no release** (inform user) |

### Override handling

If $action provides an explicit bump that is LOWER than what the changes warrant (e.g., user says `patch` but there are new features), warn the user and ask for confirmation. Do NOT silently downgrade the bump.

If $action provides a bump HIGHER than what changes warrant, proceed without warning — the user may have reasons for a larger bump.

## 7. CHANGELOG.md Format

Always follows [Keep a Changelog](https://keepachangelog.com) conventions.

### Location and Naming

Always `CHANGELOG.md` (uppercase) in the project root — the directory at the project path. This is non-negotiable regardless of project type or conventions.

### Structure

```markdown
# Changelog

All notable changes to this project are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com),
and this project adheres to [Semantic Versioning](https://semver.org/).

## [Unreleased]

## [1.2.0] - 2025-03-15

### Added
- Add streaming SSE support for chat interface
- Add Monte Carlo forecasting to portfolio view

### Fixed
- Fix infinite scroll spinner persisting after last page loaded

## [1.1.0] - 2025-02-01

### Added
- Add portfolio tracking with performance charts
```

### Writing Rules

- One bullet per **logical change** — squash related commits into one entry
- Start each bullet with an imperative verb: "Add", "Fix", "Remove", "Change", "Update", "Deprecate"
- Keep bullets concise but specific — mention feature names, endpoint names, file names where helpful
- Omit categories with no entries (do not add empty `### Fixed` sections)
- Newest release at the top, just below the header and `[Unreleased]` section
- `[Unreleased]` section is always present, even if empty after stamping a release
- Date format: ISO 8601 (`YYYY-MM-DD`)
- Version in brackets: `## [1.2.0] - 2025-03-15`

### Creating CHANGELOG.md

If CHANGELOG.md does not exist, create it with the header shown above before adding the first entry.

### Stamping a Release

When doing a full release (not draft):
1. Move all content from `## [Unreleased]` into a new versioned section
2. Leave `## [Unreleased]` empty (but present) at the top
3. Add the version number and today's date

## 8. Git Conventions

### Commit

After updating the version file and CHANGELOG.md, create a single commit:

```
chore(release): <project-name> v<version>
```

Stage ONLY these files:
- The version file (package.json, VERSION, Cargo.toml, etc.)
- CHANGELOG.md

Do NOT stage any other files. The release commit should contain only release metadata.

### Tag

Create an annotated git tag immediately after the commit:

**Monorepo sub-project:**
```bash
git tag -a <project-name>-v<version> -m "Release <project-name> v<version>"
```

**Standalone project:**
```bash
git tag -a v<version> -m "Release v<version>"
```

## 9. User Confirmation

The release skill MUST use AskUserQuestion to confirm before executing irreversible actions.

### Confirmation Point — Before Release

Present:
- **Current version** → **New version**
- **Bump type** and why (or that it was explicitly requested)
- **Classified changes** — the full changelog entry that will be written
- **What will happen** — files modified, commit message, tag name

Include options to:
- Proceed with the release
- Change the bump type
- Edit the changelog manually first
- Abort

### Confirmation Point — Post-Release Actions

If the generated skill includes post-release steps (push, deploy, publish), confirm separately before each irreversible action. The commit + tag is one confirmation. Push/deploy is another.
